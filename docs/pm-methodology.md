## 1. 用户故事（User Story）写法

### 标准格式

```
作为 [角色]，我希望 [功能/行为]，以便 [获得的价值/目的]
```

**英文版：** As a [role], I want [feature], so that [benefit].

### 示例

| 用户故事 | 分析 |
|---------|------|
| 作为**评测工程师**，我希望**一键创建评测任务并选择目标硬件**，以便**快速启动模型性能评测** | ✅ 角色明确、功能具体、价值清晰 |
| 作为**平台管理员**，我希望**查看所有评测任务的实时状态**，以便**及时发现异常并干预** | ✅ 好的用户故事 |
| 作为**用户**，我希望**系统好用**，以便**提高效率** | ❌ 角色模糊、功能不具体、价值空泛 |

### 写好用户故事的检查清单

- [ ] 角色是否具体？（避免"用户"，用"评测工程师"、"平台管理员"等）
- [ ] 功能是否可操作？（有明确的动作和对象）
- [ ] 价值是否真实？（解决了什么痛点，而非空话）
- [ ] 是否从用户视角出发？（不是技术实现视角）

## 2. INVEST 原则

每个用户故事/Issue 都应满足 INVEST 原则：

| 原则 | 英文 | 含义 | 检查方法 |
|------|------|------|----------|
| **I** | Independent | 独立的 | 能否独立开发、测试、交付？不依赖其他未完成的故事？ |
| **N** | Negotiable | 可协商的 | 实现方式是否可以讨论？不是死板的技术规格 |
| **V** | Valuable | 有价值的 | 对用户或业务是否有明确价值？ |
| **E** | Estimable | 可估算的 | 开发团队能否估算工作量？需求是否足够清晰？ |
| **S** | Small | 小的 | 能否在一个 Sprint（1-2周）内完成？太大需要拆分 |
| **T** | Testable | 可测试的 | 是否有明确的验收标准？QA 能写测试用例吗？ |

### 实操建议

- **I - 独立性**：如果两个故事必须同时上线才有意义，考虑合并或重新拆分
- **N - 可协商**：Issue 描述目标和约束，不要规定具体代码实现
- **V - 有价值**：每个故事都应该能回答"这个做完，用户能干什么新的事？"
- **E - 可估算**：如果无法估算，说明需求不清晰，需要 Spike（技术调研）
- **S - 小**：超过 8 个故事点的 Story 应拆分；超过 3 天的 Task 应拆分
- **T - 可测试**：验收标准用 Given-When-Then 格式写，或明确的 checklist

## 3. 需求拆解方法

### 拆解层级

```
PRD（产品需求文档）
 └── Epic（史诗）— 大的业务目标/模块
      └── Feature（特性）— 一组相关的功能
           └── Story（用户故事）— 单个用户价值
                └── Task（开发任务）— 具体技术实现
```

### 各层级说明

| 层级 | 粒度 | 交付周期 | 负责人 | 示例 |
|------|------|----------|--------|------|
| **PRD** | 产品级 | 1个里程碑 | PM | 《AI 硬件评测平台 PRD v1.0》 |
| **Epic** | 模块级 | 1-3个月 | PM | 评测任务管理系统 |
| **Feature** | 功能级 | 2-4周 | PM + Tech Lead | 评测任务创建与配置 |
| **Story** | 场景级 | 1-2周 | 开发 | 作为评测工程师，我希望选择评测模型和硬件目标 |
| **Task** | 实现级 | 1-3天 | 开发 | 实现模型选择下拉接口 + 前端组件 |

### 拆解示例

```
Epic: 评测任务管理
├── Feature: 评测任务创建
│   ├── Story: 创建基础评测任务（选模型、选硬件、配参数）
│   │   ├── Task: 后端 - 任务创建 API
│   │   ├── Task: 前端 - 任务创建表单
│   │   └── Task: 前端 - 模型/硬件选择器组件
│   ├── Story: 评测任务模板（保存/加载常用配置）
│   └── Story: 批量创建评测任务
├── Feature: 评测任务调度
│   ├── Story: 任务队列管理（优先级排序）
│   └── Story: 资源感知调度（根据 CPU 可用资源分配）
├── Feature: 评测任务监控
│   ├── Story: 实时查看任务执行状态
│   └── Story: 任务执行日志实时流
└── Feature: 评测任务控制
    ├── Story: 手动终止运行中的任务
    └── Story: 失败任务自动/手动重试
```

### 拆解原则

1. **自顶向下**：PRD → Epic → Feature → Story → Task
2. **MECE 原则**：相互独立、完全穷尽（不重叠、不遗漏）
3. **用户价值驱动**：每个 Story 交付后用户能感知到价值
4. **垂直切片**：Story 尽量包含前后端，而非水平切（纯前端/纯后端）
5. **合理粒度**：Story 控制在 1-2 周，Task 控制在 1-3 天

## 4. Issue 质量标准

### 一个好的开发 Issue 应包含

```markdown
## 背景与目标
为什么要做这个？解决什么问题？属于哪个 Epic/Feature？

## 用户故事
作为 [角色]，我希望 [功能]，以便 [价值]

## 功能点
- [ ] 功能点 1：具体描述
- [ ] 功能点 2：具体描述
- [ ] 功能点 3：具体描述

## 验收标准（Acceptance Criteria）
- [ ] AC1: Given [前置条件], When [操作], Then [预期结果]
- [ ] AC2: ...
- [ ] 性能要求：接口响应时间 < 500ms
- [ ] 异常处理：XX 情况下显示 XX 错误提示

## 优先级
P0/P1/P2/P3 + 说明理由

## 关联与依赖
- 依赖：#XX（XX 功能需要先完成）
- 被依赖：#YY 依赖本 Issue
- 关联 PRD 章节：第 X.Y 节

## 技术方案建议（可选）
- 建议使用 XX 技术/框架
- 数据模型参考：...
- 注意事项：...

## 设计稿（如有）
- Figma 链接 / 截图

## 里程碑
第一期（CPU 系统）- 2026.09
```

### Issue 质量检查矩阵

| 维度 | ✅ 合格 | ❌ 不合格 |
|------|---------|----------|
| **背景** | 说明了业务场景和目标 | 只有标题，没有上下文 |
| **功能点** | Checkbox 列表，每项可独立验证 | 大段文字描述，不可拆分 |
| **验收标准** | 可量化、可测试（数字/行为/状态） | "系统应该好用" "性能良好" |
| **优先级** | 有明确等级和理由 | 无优先级标注 |
| **依赖关系** | 标注了前置和后置依赖 | 未提及任何关联 |
| **技术建议** | 给出实现方向或约束 | 完全没有技术参考 |
| **粒度** | 1-2 周可完成 | 一个 Issue 包含整个模块 |

## 5. 覆盖度检查方法

### 需求追踪矩阵（RTM）

建立 PRD 需求点 ↔ Issue 的映射表：

| PRD 需求编号 | 需求描述 | 对应 Issue | 覆盖状态 |
|-------------|---------|-----------|----------|
| REQ-001 | 评测任务创建 | #36, #37 | ✅ 已覆盖 |
| REQ-002 | 评测报告生成 | — | ❌ 遗漏 |
| REQ-003 | 用户认证 | #40 | ⚠️ 部分覆盖 |

### 覆盖度检查步骤

1. **列出 PRD 所有需求点**
   - 从 PRD 中提取所有功能需求（按模块分组）
   - 每个需求点编号（REQ-XXX）

2. **建立映射**
   - 对每个 Issue，标注它覆盖了哪些 PRD 需求点
   - 对每个 PRD 需求点，标注有哪些 Issue 对应

3. **识别遗漏**
   - 没有任何 Issue 对应的需求点 → **遗漏**
   - 只有部分 Issue 覆盖的需求点 → **部分覆盖**，需检查缺什么

4. **识别冗余**
   - 不对应任何 PRD 需求点的 Issue → **超范围**，需确认是否需要

5. **定期复查**
   - 每个 Sprint Planning 前复查一次覆盖度
   - PRD 变更时立即更新映射表

### 覆盖度公式

```
覆盖率 = 已有 Issue 覆盖的需求点数 / PRD 总需求点数 × 100%
```

**目标：第一期 Issue 创建完成时，覆盖率应达到 100%。**

### 实操工具建议

- **GitHub Projects**：用 Board 视图按 Epic 分组，快速发现空白区域
- **GitHub Labels**：给 Issue 打 Epic 标签（如 `epic:任务管理`），按标签过滤检查
- **Milestone**：所有第一期 Issue 归入同一 Milestone，方便追踪进度
