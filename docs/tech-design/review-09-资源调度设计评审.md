# 资源调度设计评审反馈

> 评审人：麦克雷（PM/测试）| 日期：2026-04-12 | 评审文档：resource-scheduling-design.md v1.0

---

## 总体评价

文档质量不错，场景梳理清晰，从用户视角出发是对的。5个典型场景覆盖了主要用例，4个模块拆分合理。但有几个架构层面的问题需要讨论清楚再动手，否则后面返工成本很高。

**评分：7/10** — 场景分析和数据模型设计扎实，但存在过度设计风险和几个关键设计缺陷。

---

## 🔴 必须修改（阻塞实施）

### 1. RunSpec 预置过于死板，应该用"需求描述"而不是"固定规格"

**问题：** 当前 RunSpec 设计成固定的 9 种系统预置（1n1g, 1n4g, 2n8g...），但实际场景中：
- 不同芯片的卡数不一样（有的机器 4 卡，有的 8 卡，有的 16 卡）
- 预置 `1n8g` 在 4 卡机器上根本无法使用
- 用户应该描述"我需要多少卡"，而不是"我要用哪个固定规格"

**建议：** RunSpec 改为**需求描述式**：
```
{
  "nodeCount": 2,        // 我需要几台机器
  "gpuPerNode": null,    // null = 用该节点所有卡; 4 = 指定每节点4卡
  "exclusive": true,     // 是否独占
  "minGpuTotal": 16      // 最少总 GPU 数（调度器自动匹配）
}
```

这样用户说"我要 16 张卡"，调度器自己决定是 2×8 还是 4×4，而不是用户必须知道集群里机器是几卡的。

### 2. Phase 分期有问题：Phase 1 应该只做 RunSpec + 资源池，不碰 GPU Slot

**问题：** Phase 1 的 checklist 里已经勾掉了芯片亲和性（已完成），但剩余的"RunSpec 表 + CRUD + 前端运行规格选择步骤"和后面几个 Phase 之间有隐式依赖。

**建议重新划分：**
- **Phase 1（1周）**：RunSpec 数据模型 + API + 前端选择 UI + 创建 Plan 时关联 RunSpec。调度层先不改，还是按现有单任务调度跑，但数据结构就位了。
- **Phase 2（2周）**：资源池重构 + GPU Slot + 单机多卡调度 + Agent 改造
- **Phase 3（3周）**：多机多卡（这个依赖 Phase 2 的 GPU Slot）
- **Phase 4（独立）**：K8s，可以和 Phase 2/3 并行

### 3. GPU Slot 表的并发问题没有设计

**问题：** 多个调度器实例（或定时任务 + 手动触发）同时执行时，两个任务可能分配到同一张 GPU。当前设计只有 `status` + `allocated_task_id`，没有乐观锁或分布式锁。

**建议：** 
- `gpu_slots` 表加 `version` 字段做乐观锁（和现在 task 表一样）
- 分配 GPU 时用 `UPDATE ... WHERE status = 'FREE' AND version = ?`
- 或者用 `SELECT ... FOR UPDATE` 悲观锁
- 这是并发安全的基础，必须在设计文档中明确

## 🟡 建议修改（不阻塞但强烈建议）

### 4. 调度策略过早引入，先做好 least_loaded 就行

4 种调度策略（least_loaded / round_robin / pack / spread）在当前阶段没有实际用户需求。我们现在总共 2 个节点，不需要策略选择。

**建议：** Phase 1-2 只实现 `least_loaded`，其他策略留接口但不实现。避免测试矩阵爆炸。

### 5. K8s 设计太粗糙，要么砍掉要么单独出文档

场景 5 和 Phase 4 的 K8s 部分写得很概要：
- 没有说清楚用 Kubeflow PyTorchJob 还是原生 Job
- 没有 GPU 资源配额（ResourceQuota）设计
- 没有 Pod 异常处理（OOM、Eviction）
- K8s 和 bare-metal 的报告合并逻辑没设计

**建议：** K8s 部分从这个文档里拆出去，作为 Phase 4 的单独设计文档。当前文档聚焦 bare-metal 场景。

### 6. 多机任务的错误处理设计缺失

文档提到了风险"某个节点掉线，整个任务如何处理"，但没有给出设计。这是多机调度的核心难题。

**至少要明确：**
- 任意一个节点失败 → 整个任务标记 FAILED？还是重试该节点？
- Master 节点挂了怎么办？
- 任务完成后 GPU Slot 回收的时序（先回收再更新 task 状态？还是反过来？）

### 7. 前端 UI 设计中"降级规格"按钮需要谨慎

QUEUED 状态下的 `[降级规格（改为 2 卡）]` 按钮：
- 降级意味着测试条件变了，评测结果不可比
- 用户可能误操作
- **建议改为**：显示"修改运行规格"入口，但要有明确警告"修改规格将导致评测条件变化"

## 🟢 认可的部分

1. **用户场景驱动设计** — 先列 5 个场景再设计模块，方向正确
2. **芯片亲和性作为硬约束** — "宁可排队不跑错"的原则已经在代码中验证通过
3. **数据模型设计清晰** — SQL schema 定义详细，字段命名规范
4. **Agent 协议扩展向后兼容** — 新增 `runSpec` 和 `distributed` 字段，老 Agent 不受影响
5. **报告标注执行环境** — 这个已经在亲和性迭代中实现了一部分，方向对
6. **task_node_allocations 表** — 多机任务中间表设计合理，可以追踪每个节点的执行状态

## 实施建议

1. **先验证 Phase 1 的数据模型**：创建 RunSpec 表，插入预置数据，确认 API 可用。这一步不碰调度器，风险最低。
2. **Phase 2 是核心也是最难的**：GPU Slot 的并发安全 + Agent 改造（上报 GPU 列表、CUDA_VISIBLE_DEVICES），建议预留充足测试时间。
3. **每个 Phase 完成后我做端到端测试验证**，和这次亲和性调度测试一样，通过 API + 数据库验证调度结果。
4. **K8s 不急**，先把 bare-metal 的多卡调度跑通再说。
