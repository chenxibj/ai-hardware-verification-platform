# 评测任务调度与分发技术方案

> 版本：v1.0 | 更新日期：2026-04-06 | 作者：菜菜子

## 1. 问题背景

当前系统在用户创建评测任务并点击"启动"后，后端仅将任务状态改为 RUNNING，**不会自动将子任务分发到计算节点 Agent 执行**。用户看到所有测试项停留在"排队中"（PENDING），需要手动调用 Agent API 才能触发执行。

**期望行为：** 用户点击启动 → 系统自动逐个分发子任务到节点 → 实时更新进度 → 全部完成后自动生成报告。

## 2. 架构设计

### 2.1 链式分发模型

Agent 是单任务串行执行的（一次只能跑一个测试），因此采用**链式分发**而非并行分发：

```
用户点击"启动"
    │
    ▼
startPlan()
    ├── 1. Plan 状态 → RUNNING
    ├── 2. 分发第 1 个 PENDING 子任务 → Agent
    │       ├── Task 状态 → RUNNING
    │       └── POST http://agent:8090/execute
    │
    ▼ (Agent 执行完毕，回调)
TaskCompleteController.completeTask()
    ├── 3. Task 状态 → COMPLETED
    ├── 4. 更新 Plan 进度 (completedTasks++, progress%)
    ├── 5. 还有 PENDING 任务？
    │       ├── YES → 分发下一个 PENDING 子任务 → Agent (回到步骤 2)
    │       └── NO  → Plan 状态 → COMPLETED
    │                  └── 自动生成评测报告
    │
    ▼
完成
```

## 3. 核心组件

### 3.1 TaskDispatcher（任务分发器）

**位置：** `backend/src/main/java/com/lab/plan/TaskDispatcher.java`

**职责：**
- 接收分发请求（planId）
- 从 PENDING 任务中取第一个
- 构造 Agent /execute 请求（taskId + evalType + params）
- 发送到目标节点 Agent
- 处理分发失败（标记 task 为 FAILED，尝试下一个）

**关键方法：**

| 方法 | 说明 |
|------|------|
| `dispatchNextTask(planId)` | 查找并分发计划中的下一个 PENDING 任务 |
| `dispatchSingleTask(task, agentUrl, plan)` | 将单个任务发送到 Agent |

**Agent URL 解析逻辑：**
```
node.ipAddress + ":" + node.agentPort
如果是 Docker 内网 IP (172.x / 10.x) → 替换为 localhost
```

### 3.2 调用入口

| 调用点 | 时机 | 说明 |
|--------|------|------|
| `EvaluationPlanService.startPlan()` | 用户点击启动 | 触发首次分发 |
| `TaskCompleteController.completeTask()` | Agent 回报结果 | 链式分发下一个 |
| `EvaluationPlanService.resumePlan()` | 暂停后恢复 | 继续分发 PENDING 任务 |

### 3.3 分发请求格式

**POST http://{agent_host}:{agent_port}/execute**

请求体：
```json
{
  "taskId": 123,
  "evalType": "OPERATOR",
  "params": {
    "operator": "MatMul",
    "dtype": "FP32",
    "shape": "Medium"
  },
  "config": {
    "operator": "MatMul",
    "dtype": "FP32",
    "shape": "Medium"
  }
}
```

请求头：
```
Content-Type: application/json
X-Agent-Token: <shared-secret>
```

响应（Agent 侧）：
- 200：任务已接受
- 409：节点繁忙（正在执行其他任务）
- 401：认证失败

### 3.4 异步执行

TaskDispatcher 使用 `@Async` 注解，分发操作在独立线程中执行，不阻塞 HTTP 响应。

需要在 Spring Boot 应用中启用 `@EnableAsync`。

## 4. 错误处理

### 4.1 分发失败

| 场景 | 处理 |
|------|------|
| 节点不存在 | 记录日志，Plan 保持 RUNNING（等待人工干预） |
| 节点 OFFLINE | 记录日志，不分发（心跳恢复后可手动重试） |
| Agent 返回 409（繁忙） | 等待 5 秒后重试一次，仍失败则跳过（等回调触发下一轮） |
| Agent 连接超时 | 标记当前 task 为 FAILED，继续分发下一个 |
| Agent 返回 500 | 标记当前 task 为 FAILED，继续分发下一个 |

### 4.2 任务失败不阻塞

单个任务失败不阻塞整个计划。分发器会跳过失败任务，继续分发后续 PENDING 任务。

### 4.3 全部任务完成判定

在 TaskCompleteController 中：
```
completedCount = COMPLETED + FAILED 任务数
allDone = completedCount >= totalTasks
```

## 5. 暂停 / 取消

| 操作 | 影响 |
|------|------|
| 暂停 (pause) | Plan→PAUSED，不再分发新任务。已在 Agent 执行中的任务继续完成 |
| 恢复 (resume) | Plan→RUNNING，触发 `dispatchNextTask` 继续分发 |
| 取消 (cancel) | Plan→CANCELLED，不再分发。PENDING 任务保持原状。正在执行的任务继续完成 |

## 6. 日志采集

### 6.1 Agent 侧

- 评测脚本的 stdout/stderr 写入 `agent/logs/{taskId}.log`
- 结果上报时，日志内容作为 `logs` 字段附带

### 6.2 平台侧

- TaskCompleteController 收到结果后：
  - 将日志写入 `task_logs` 表
  - 前端通过 `GET /api/tasks/{taskId}/logs` 查询
  - 支持 `GET /api/tasks/{taskId}/logs/download` 下载

### 6.3 实时日志（后续规划）

当前为任务完成后一次性上报日志。后续可改为：
- Agent 执行期间定时推送日志（WebSocket 或 SSE）
- 前端实时刷新展示
