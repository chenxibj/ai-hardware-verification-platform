## ⚠️ 需要补充/修正的问题

### 1. [P0] 缺少「贡献管理」设计

**问题:** 原始 PRD 2.4 明确提到「资产分享与权限控制」涉及贡献管理的概念（谁贡献了哪些资产、贡献排行等），当前 PRD 只有复用统计，缺少「贡献者」视角的管理和展示。

**建议:**
- 增加「贡献者」维度：资产详情页展示贡献者信息，资产列表支持按贡献者筛选
- 增加「贡献排行榜」：个人贡献统计（上传资产数、被复用次数、被下载次数）
- 与社区模块联动：优质贡献者可在社区中展示

### 2. [P0] 缺少与现有 RBAC 体系的集成说明

**问题:** 原始 PRD 模块 4 有完整的 RBAC 体系（超级管理员、平台管理员、租户管理员、评测专员、资产管理员、普通用户），资产管理的权限控制需要与之集成，当前 PRD 没有说明如何对接。

**建议:**
- 明确各角色对资产模块的权限映射：
  - 超级管理员/平台管理员：全量管理权限（含分类配置、回收站永久删除、备份恢复）
  - 租户管理员：本租户资产管理权限（含分享审核）
  - 资产管理员：资产 CRUD + 版本管理 + 分享管理
  - 评测专员：资产查看 + 复用权限
  - 普通用户：个人资产 CRUD + 公共资产查看/复用
- 在 API 层增加权限注解说明（哪些接口需要哪些角色）

### 3. [P1] 与评测任务的联动交互不够深

**问题:** 原始 PRD 强调「资产复用」要与评测任务深度联动，当前 PRD 虽然提到了，但 API 层面只有一个 `/api/assets/selectable` 预留接口，缺少具体的联动交互设计。

**建议:**
- 补充评测任务创建表单中「资产选择器」组件的详细交互设计
- 明确资产选择器的筛选维度（类型、分类、标签、版本）
- 增加「临时上传 + 同步保存到资产库」的选项
- 补充评测任务与资产的关联关系表设计

### 4. [P1] Phase 1 压缩包自动解压应推迟到 Phase 2

**问题:** 批量上传压缩包自动解压功能实现复杂度高（识别文件类型、处理嵌套目录、错误处理），放在 Phase 1 会增加交付风险。

**建议:**
- Phase 1 只做多文件并行上传（前端选择多个文件 + 逐个上传）
- 压缩包自动解压移到 Phase 2
- 减轻 Phase 1 工作量，确保核心 CRUD 链路稳定

### 5. [P1] 分享范围粒度偏粗

**问题:** 原始 PRD 提到「精细化权限配置」，当前设计的 PRIVATE/TEAM/PUBLIC 三种粒度在实际使用中不够灵活。

**建议:**
- Phase 2 增加「指定用户分享」（选择具体用户 ID 列表）
- Phase 2 增加「指定角色分享」（如仅对评测专员可见）
- Future 增加「分享审批流程」（全平台共享前需管理员审批）

### 6. [P2] 版本号策略过于简单

**问题:** v{major}.{minor} 两级版本号在实际使用中可能不够（无法区分 bugfix 和 feature 级别的更新），且明确禁止自定义版本号过于限制。

**建议:**
- 支持三段式语义化版本 v{major}.{minor}.{patch}
- 或至少允许用户在上传新版本时自定义版本号（需校验格式和唯一性）
- 默认自增逻辑保留作为推荐值

### 7. [P2] 缺少存储配额管理

**问题:** 原始 PRD 2.4.6 提到存储监控，当前 PRD 有存储用量统计但没有配额管理。对于多租户场景，配额管理是刚需。

**建议:**
- Phase 2 增加租户级存储配额设置（管理员配置每个租户的存储上限）
- 用户级配额（可选，由租户管理员配置）
- 超配额阻止上传 + 配额预警（80% 提醒，90% 告警）

### 8. [P2] 中文搜索方案需明确

**问题:** MVP 用 PostgreSQL 全文检索合理，但给了「100 万资产内 500ms」的性能承诺。PG tsvector 在中文分词上表现一般，默认 simple 配置无法有效分词中文。

**建议:**
- 明确标注需要配置 zhparser 或 pg_jieba 中文分词插件
- 或在技术方案阶段选型确认
- 性能承诺改为「10 万资产内 500ms（基于 PG 全文检索）」，100 万级需 ES

### 9. [P2] 回收站定时清理缺少技术方案

**问题:** MinIO lifecycle rule 设 30 天 OK，但 PostgreSQL 中的记录清理需要定时任务，当前未说明实现方式。

**建议:**
- 补充定时清理方案：Spring @Scheduled 或 Quartz 定时任务
- 每日凌晨执行：清理 deleted_at < NOW() - INTERVAL '30 days' 的记录
- 清理前先删除 MinIO 文件，再删除 PG 记录（保持一致性）
- 记录清理日志

### 10. [P2] 资产校验分阶段可优化

**问题:** Phase 1 只有基础校验（文件大小/格式/SHA-256），高级校验（模型可加载、数据集完整性）全部放到 Phase 3。

**建议:**
- Phase 2 增加「模型可加载」基础检查（ONNX Runtime 尝试加载，≤30s 超时）
- Phase 3 做完整校验（数据集抽样、敏感数据扫描等）
- 理由：模型能否正常加载是核心使用场景，不应推到最后
