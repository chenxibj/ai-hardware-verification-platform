# 算子性能与精度评测 PRD v1.0 评审反馈

> **评审人:** 麦克雷（产品经理）  
> **评审日期:** 2026-04-08  
> **评审文档:** docs/product-design/operator-eval-design.md (v1.0, 1577行)  
> **评审结论:** 方法论扎实、指标体系完整，是目前写得最好的一份 PRD。核心问题在于落地路径——需要更清晰地区分"我们现在能做的"和"需要芯片厂商配合才能做的"。

---

## 总体评价

这份 PRD 在算子评测方法论上做得非常扎实：

- 精度评测三种方法互补（参考对比 + 跨精度 + 数学性质），逻辑自洽
- 9 个精度指标 + 阈值体系参考了 DeepLink / ONNX RT / IEEE 754，不是拍脑袋
- 性能指标从 5 个扩到 14 个，补了 GFLOPS / 算力利用率 / 延迟 CV 等关键指标
- 对比报告设计得很完整，三种维度 + 四种图表 + tradeoff 分析

主要需要调整的是实施优先级和与现实约束的匹配。

---

## 🔴 必须修改

### 1. 算子清单和现有代码能力不匹配，需要标注哪些是"现有"哪些是"待开发"

**问题：** PRD 列了 23 个算子，但现有 `cpu_operator_benchmark.py` 只覆盖 14 个，且都是纯 CPU 的 NumPy 实现。PRD 里的 Conv3D、DepthwiseConv、RMSNorm、SiLU、Embedding、以及 3 个复合算子、4 个通信算子在现有代码中都不存在。

**建议：**
- 算子清单分两列标注：✅ 已有实现 / 🆕 需新增
- Phase 1 只做已有实现的算子的精度增强，不要同时新增算子 + 新增精度
- 新增算子放 Phase 2，降低 Phase 1 的风险

### 2. FP16/BF16/INT8 多精度支持的现实约束没写清楚

**问题：** PRD 把多精度（FP32/FP16/BF16/INT8）作为 Phase 1 P0 任务，工作量写 2d。但现有环境是纯 CPU（Intel Xeon），NumPy 对 FP16 支持有限（没有原生 FP16 计算，只能 cast），BF16 在 NumPy < 1.24 不支持，INT8 量化更需要额外框架（如 ONNX Runtime 的量化 API）。

**建议：**
- Phase 1 的多精度限定为 **FP32 + FP16**（NumPy 可 cast 模拟，够验证精度流程）
- BF16 需要确认 NumPy 版本（现有环境 numpy 1.24.0 刚好支持 bfloat16 dtype，但没有原生计算支持）
- INT8 量化涉及 scale/zero_point 的校准流程，复杂度远超 cast，建议推到 Phase 2
- 在 PRD 中明确写出每种 dtype 的实现方式和依赖

### 3. `peak_gflops` 硬编码 1200.0 不可接受，算力利用率需要自动检测或手动配置

**问题：** 性能数据结构示例里 `peak_gflops_fp32: 1200.0` 是硬编码。`compute_util_percent = actual_gflops / peak_gflops × 100` 这个指标的价值完全取决于 `peak_gflops` 的准确性。但不同芯片、不同精度的理论峰值不同，需要一个可靠的获取方式。

**建议：**
- 方案 A：芯片注册时填入理论峰值（在芯片管理模块添加 `peak_gflops_fp32/fp16/int8` 字段）
- 方案 B：Agent 端通过 STREAM benchmark 或 `lscpu` 自动估算（不够准确但聊胜于无）
- 方案 C：如果获取不到峰值，`compute_util_percent` 留空或标记为 N/A，不要硬编码一个假数字
- **这个决策需要在 PRD 里明确**，否则开发会随意处理

### 4. Phase 1 工作量严重低估

**问题：** Phase 1 任务清单写了 13 天工作量，要做到：精度验证 3 种方法 + 9 个精度指标 + 多精度支持 + 性能指标增强 + 2 张新表 + 结果解析服务 + 汇总 API + 对比 API。这对一个从零建设精度验证能力的项目来说太紧了。

**建议：**
- Phase 1 砍掉对比 API（`POST /reports/compare`），推到 Phase 2
- Phase 1 精度验证只做方法 1（参考对比）和方法 3（数学性质），方法 2（跨精度对比）依赖多精度支持做完后再做
- 调整后的 Phase 1 聚焦：**性能指标增强 + FP32 精度验证 + 数据存储**，大约 2 周合理

### 5. 精度阈值需要标注"初始值，可调整"

**问题：** 4.3 节的阈值表写得很具体（FP32 max_abs_error ≤ 1e-5，FP16 cosine_similarity ≥ 0.999），但这些值在实际跑起来之前无法确认是否合理。特别是 INT8 的阈值（cosine_similarity ≥ 0.95），不同量化方式差异很大。

**建议：**
- 所有阈值标注为"初始推荐值"
- 增加配置化机制：阈值存在 application.yml 或数据库中，不硬编码在 Agent 脚本里
- Phase 1 跑通后根据实际数据调整阈值，Phase 2 出正式版
- 补充说明：4.3.2 的"算子特定阈值覆盖"机制很好，但需要明确优先级（算子特定 > 通用 > 默认）

## 🟡 建议优化

### 6. 对比报告的自动推荐文字（recommendation）不建议 Phase 1 做

5.2 节示例里有 `"recommendation": "Ascend 910B 在计算密集型算子上表现显著优于 Intel Xeon..."`。这种自然语言生成逻辑复杂（需要判断哪些算子快、快多少、分什么类别总结），容易出错。

**建议：** Phase 1/2 只提供数据和涨跌标识，recommendation 字段留空或写固定模板。自然语言总结可以后续用 LLM 生成，不要手写规则。

### 7. 前端图表选型建议补充技术方案

5.4 节列了 5 种图表（柱状图/雷达图/热力图/折线图/表格），但没说用什么库。

**建议：** 明确使用 ECharts（React 项目用 echarts-for-react），和现有前端技术栈保持一致。如果现有项目已经用了某个图表库，优先复用。

### 8. Agent 脚本命名建议统一规范

现有 `cpu_operator_benchmark.py` 要升级为 `operator_benchmark_v2.py`。这个命名不太好——以后 v3 呢？

**建议：**
- 脚本名不带版本号：`operator_benchmark.py`
- 版本通过脚本内部变量 `SCRIPT_VERSION = "2.0"` 管理
- 上报数据的 `version: "2.0"` 字段保留，用于后端识别数据格式

### 9. 数学性质验证的 tolerance 需要区分 dtype

4.4 节的数学性质检查表，例如 Softmax 的 `abs(sum(output, dim) - 1.0) ≤ 1e-6`，这个 1e-6 的 tolerance 只对 FP32 合理。FP16 下 Softmax 输出和可能偏差到 1e-3。

**建议：** 数学性质的 tolerance 也按 dtype 分级，类似 4.3.1 的通用阈值表。

### 10. 缺少"评测结果打分"机制

PRD 定义了 PASS/WARNING/FAIL 三级判定，但没有**综合评分**——用户想知道"这个芯片的算子能力总体打几分？"。

**建议（P2 考虑）：**
- 定义一个 0-100 的综合评分公式，例如：`score = w1 * perf_score + w2 * accuracy_pass_rate + w3 * coverage`
- 每个算子有单独评分，汇总为芯片总分
- 对比报告可以直接比总分
- 这个不急，但 PRD 里可以留个口子

## 🟢 写得好的地方

1. **精度评测三种方法的设计是真正有技术深度的产品设计**——方法 1（参考对比）解决"正确性"，方法 2（跨精度）解决"量化损失"，方法 3（数学性质）解决"快速冒烟"。三者互补，逻辑严密。

2. **4.4 各算子精度验证要点** 是整篇 PRD 最有价值的部分。每个算子的特殊验证点和边界测试用例写得非常具体（Softmax 的大值稳定性、MatMul 的对称性、BatchNorm 的 running stats 累积误差），开发拿来就能写代码。

3. **阈值体系有据可查**——FP32 对标 IEEE 754 精度理论，FP16/BF16 参考 DeepLink，INT8 参考 ONNX RT conformance。不是拍脑袋的数字。

4. **测试配置矩阵的分层策略**很务实——1840 个全量组合缩减到快速 10 / 标准 180 / 全面 1520，给了用户选择权。

5. **FLOPs 计算公式表**（附录 A）直接可用，每个算子的计算量公式都给了，Agent 开发不用再查论文。

6. **对比报告的 JSON 结构设计得很清晰**——baseline/target 对称、delta 独立计算、summary 自动汇总，前端拿到数据就能渲染。

## 修改后的实施优先级建议

### Phase 1（2周）—— 精度验证基础 + 性能增强

| 任务 | 说明 |
|------|------|
| Agent: 现有 14 个算子的性能指标增强 | 补 GFLOPS / 算力利用率 / 内存占用 / 延迟 CV / warmup overhead |
| Agent: FP32 精度验证（方法 1 + 方法 3） | 参考实现对比 + 数学性质验证，暂不做跨精度 |
| Agent: FP16 支持（cast 模拟） | 仅 cast 级别，验证精度流程 |
| 后端: 2 张新表 + 结果解析 | operator_perf_results + operator_accuracy_results |
| 后端: 精度汇总 API | /accuracy-results/summary |
| 峰值算力: 确定获取方案 | 芯片注册字段 or 自动检测 or N/A |

### Phase 2（2周）—— 跨精度 + 对比 + 前端

| 任务 | 说明 |
|------|------|
| Agent: 跨精度对比（方法 2） | FP32 vs FP16 精度损失量化 |
| Agent: 新增算子（Conv3D/RMSNorm/SiLU/Embedding/融合算子） | 扩充算子覆盖 |
| 后端: 对比报告 API | POST /reports/compare + GET |
| 后端: Precision-tradeoff API | 精度-性能权衡分析 |
| 前端: 评测结果详情页 | 性能 + 精度双标签 |
| 前端: 对比报告页 | 柱状图 + 详细表格 + 热力图 |

### Phase 3（2周）—— GPU + 多芯片 + 高级功能

- GPU 适配、多芯片对比大屏、通信算子、综合评分（保持原计划）
