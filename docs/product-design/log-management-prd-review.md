# 日志管理 PRD v1.0 评审反馈

> **评审人:** 麦克雷（产品经理）  
> **评审日期:** 2026-04-07  
> **评审文档:** docs/product-design/log-management-prd.md  
> **评审结论:** 基本可用，需调整实现路径

---

## 总体评价

PRD 结构清晰，问题分析准确（假数据、无持久化、日志太简单三个核心问题都抓到了），现状盘点做得好。主要问题在实现路径上——P0 塞了太多东西，存储方案过度设计。

---

## 🔴 必须修改

### 1. P0 需要更聚焦，拆掉 WebSocket

**问题：** P0 包含了 WebSocket 实时推送，但项目从没做过 WebSocket。光调通 Spring WebSocket + 前端连接 + 重连 + 心跳就至少 1.5 天，P0 的 3 天估时会爆。

**建议：**
- P0 用 **HTTP 轮询**（2s 间隔 + `after` 游标）代替 WebSocket，完全够用
- WebSocket 推到 P1
- P0 应该是：移除假数据 → 对接已有 REST API → 日志级别颜色 → 导出对接
- 这样 P0 真正能 3 天搞定

### 2. US-6（故障定位）和 US-7（失败后查日志）应该在 P0

**问题：** 任务失败后看最后日志、ERROR 高亮——这是最基本的需求，PRD 里写了用户故事但实现放在了 P1/P2。

**建议：**
- ERROR 日志红色高亮、支持只看 ERROR → 移到 P0
- 任务失败时 stderr 完整保留 → 移到 P0
- 从任务列表一键跳转失败日志 → 移到 P0

### 3. 存储方案过度设计，需要简化

**问题：** 月日志量估算才 450MB，一年 5.4GB。但设计了热/温/冷三级存储 + 分区表 + gzip 归档 + `log_archives` 表 + `log_export_jobs` 表。对这个体量来说太重了。

**建议：**
- P0/P1：**就一张 `task_logs` 表 + 索引**，不要分区，不要归档
- P2 再评估是否需要归档（大概率不需要，PostgreSQL 处理几百万行毫无压力）
- 删除 `log_archives` 表和 `log_export_jobs` 表的 P0/P1 计划
- 如果未来真需要归档，到时候再加，不要提前造基建

### 4. WebSocket 推送的中间通知层缺失

**问题：** Agent 通过 HTTP POST 上报日志到后端，后端写入数据库。但 WebSocket 需要在写入后实时推送给前端——这中间需要一个通知机制（DB 写入 → WebSocket 推送），PRD 里没设计。

**建议：**
- 明确技术方案：用 Spring ApplicationEvent 或内存 pub/sub
- 写清楚流程：Agent HTTP POST → Controller 写 DB → 发布事件 → WebSocketHandler 推送
- 这个是 P1（WebSocket）的前置设计，需要补充

### 5. `sequence_no` 生成策略需要明确

**问题：** `sequence_no` 是日志排序和去重的核心字段，但 PRD 没说谁生成、怎么生成。

**建议：**
- **后端自增生成**，不要让 Agent 管（多 Agent 并发会冲突）
- 方案：每个 task_id 维度内的递增序号，用数据库 BIGSERIAL 或在 INSERT 时计算
- 需要在 PRD 里明确写出来，否则开发会卡住

## 🟡 建议优化

### 6. METRIC 日志的 sparkline 图表优先级降低

迷你图表是锦上添花，P1 重点应该是结构化存储 + 搜索过滤。建议 sparkline 推到 P2。

### 7. 日志对比的对齐逻辑不够具体

"按 batch_size / model / operator 匹配"——如果两次评测参数不完全一致怎么办？需要定义：
- 匹配不上时的行为（跳过？标红？显示为"仅任务A"？）
- 对齐算法的优先级（先按 model 匹配，还是先按 batch_size？）

否则开发时会卡在产品逻辑不清上。

### 8. Agent 端 `_classify_log_line()` 的正则需要加强

`r'.*\d+/\d+.*'` 太粗糙，会误判（比如 "error code 1/0" 也会被分类成 PROGRESS）。

**建议：**
- 让评测脚本主动输出带 tag 的格式：`[PROGRESS] 3/10 执行中...`
- Agent 端优先匹配 tag，兜底再用正则
- 或者定义一个简单的 JSON 协议让脚本输出结构化数据

### 9. 日志对比 API 建议改成 POST

`GET /api/logs/compare?taskId1=xxx&taskId2=xxx` → 改成 `POST /api/logs/compare`

理由：返回数据结构复杂，后续可能扩展到 3+ 任务对比，POST body 更灵活。

### 10. P0 工作量重新估算

按上面的调整（去掉 WebSocket，加入 ERROR 高亮和失败日志），P0 应该重新估时：

| # | 功能点 | 估时 |
|---|--------|------|
| 1 | 移除前端假数据（PlanMonitor.js） | 0.5 天 |
| 2 | 前端对接 `GET /tasks/{taskId}/logs` | 0.5 天 |
| 3 | Agent 端日志级别区分（INFO/WARN/ERROR） | 0.5 天 |
| 4 | 前端日志级别颜色渲染 + ERROR 高亮 | 0.5 天 |
| 5 | 前端对接 `/logs/download` 导出 | 0.5 天 |
| 6 | HTTP 轮询实现（2s 间隔 + after 游标） | 0.5 天 |
| **合计** | | **3 天** |

## 🟢 写得好的地方

1. **现状盘点很准确**：把已有的 26 条日志数据、Agent 的 flush 机制（5s/50行）、前端 `LOG_TEMPLATES` 假数据都摸清了，开发不用再花时间调研
2. **兼容性设计到位**：旧接口保留、新字段有默认值、渐进式迁移路径清晰
3. **数据库迁移脚本直接可用**：开发拿来跑就行，节省时间
4. **日志量估算合理**：基于实际场景推算，没有拍脑袋
5. **用户故事覆盖全面**：7 个 US 基本涵盖了所有角色的核心需求
6. **API 设计规范**：游标分页、参数化查询、批量接口都考虑到了

## 修改后的优先级建议

### P0（3天）—— 去假数据 + 持久化 + 基本可用
- 移除前端模拟日志
- 对接已有后端 API
- 日志级别区分 + 颜色渲染
- ERROR 高亮 + 失败日志展示
- HTTP 轮询（2s + 游标）
- 导出对接

### P1（5天）—— 结构化 + 搜索 + 实时
- 结构化日志格式 + 批量上报
- WebSocket 实时推送（含通知层设计）
- 搜索过滤工具栏
- 日志统计接口
- `sequence_no` 排序机制

### P2（5天）—— 分析 + 高级功能
- METRIC 日志渲染 + sparkline
- 性能数据提取接口
- 日志 ↔ 报告关联
- 任务日志对比
- 多格式导出
- 归档（如果真需要的话）
