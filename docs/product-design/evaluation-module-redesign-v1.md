## 第一部分:产品定位与目标

### 1.1 一句话定义

**AHVP 是一个以芯片为中心的 AI 硬件评测平台,通过标准化的算子测试和模型测试,生成芯片能力评价报告,帮助国产芯片厂商、框架厂商和算力采购方回答"这颗芯片适合什么"。**

关键词拆解:
- **以芯片为中心** - 所有操作围绕芯片档案展开,不是围绕"评测类型"
- **标准化** - 评测方案可复现、结果可对比
- **芯片能力评价报告** - 最终交付物,不是一堆散落的测试日志
- **回答"适合什么"** - 业务目标驱动,不是功能堆砌

### 1.2 目标用户和使用场景

| 用户角色 | 典型场景 | 核心诉求 | 使用频率 |
|---------|---------|---------|---------|
| **芯片厂商评测工程师** | 新品流片后全面验证;参加季度评测(DeepLink等);版本迭代性能回归 | 高效完成数百个测试项,生成可对外交付的评测报告 | 高(日常工作) |
| **芯片厂商产品经理** | 查看自家芯片 vs A100/H100 的性能差距;跟踪版本迭代进展 | 直观看到芯片能力画像,快速定位短板 | 中(每周查看) |
| **框架适配工程师** | 验证 PyTorch 在新芯片上的算子覆盖率和性能 | 知道哪些算子 fallback、性能差距多大 | 中 |
| **算力采购决策者** | 对比多家芯片选型 | 性能/能效/生态成熟度对比报告 | 低(项目制) |
| **内部测试团队(SenseCore)** | 新芯片引入 SenseCore 前的适配验证 | 确保芯片满足平台最低要求 | 中 |

### 1.3 与竞品的差异化定位

市场现状分析:

| 象限 | 竞品 | 特点 | 缺什么 |
|------|------|------|--------|
| **评测逻辑强 + UI 弱** | FlagPerf、AIChipBenchmark、lm-eval-harness | 算法严谨、覆盖面广 | 没有 Web 界面,结果靠 Excel/PDF,无法在线管理 |
| **产品体验好 + 平台锁定** | 阿里云 CNP、Azure AI Foundry | Web 可视化、引导式操作 | 绑定特定云,不支持跨平台 |
| **行业权威 + 不够敏捷** | MLPerf、DeepLink 季度评测 | 公信力最强 | 周期太长(半年/季度),线下流程为主 |
| **以芯片为中心** | (空白) | - | **没有一款产品真正做到"注册芯片→评测→生成芯片档案报告"的闭环** |

**AHVP 的差异化定位:行业唯一以"芯片档案"为核心的 Web 化评测管理平台。**

具体差异点:
1. **芯片档案制** - 其他工具以"测试任务"为中心,AHVP 以"芯片"为中心,所有评测结果沉淀到芯片档案
2. **Web 可视化 + 在线管理** - 弥补 FlagPerf/AIChipBenchmark 无 UI 的短板
3. **芯片能力画像** - 竞品只给原始数据,AHVP 自动生成多维雷达图和适用场景推荐
4. **版本追踪** - 同一芯片不同版本的性能演进可视化,竞品普遍缺失
5. **不绑定单一平台** - 不同于阿里云 CNP 只能用灵骏,AHVP 支持任意计算节点

## 第二部分:核心业务流程

### 2.1 以芯片为中心的主流程

```
┌─────────────────────────────────────────────────────────────────────────┐
│                        AHVP 核心业务闭环                                │
│                                                                         │
│   1 注册芯片          2 创建评测计划        3 执行评测                  │
│   ┌──────────┐       ┌──────────────┐      ┌──────────────┐            │
│   │ 芯片型号  │──────▶│ 选择评测项    │─────▶│ 自动调度执行  │            │
│   │ 厂商信息  │       │ 配置参数     │      │ 实时监控进度  │            │
│   │ 技术规格  │       │ 选择节点     │      │ 收集结果数据  │            │
│   └──────────┘       └──────────────┘      └──────┬───────┘            │
│                                                     │                   │
│   5 芯片档案沉淀      4 生成芯片评价报告            │                   │
│   ┌──────────────┐   ┌──────────────────┐          │                   │
│   │ 能力画像更新  │◀──│ 能力雷达图        │◀─────────┘                   │
│   │ 历史数据归档  │   │ 适用场景推荐      │                              │
│   │ 版本对比基线  │   │ 瓶颈分析和优化建议 │                              │
│   └──────────────┘   └──────────────────┘                              │
└─────────────────────────────────────────────────────────────────────────┘
```

**为什么是这个流程而不是现有的"创建任务→看结果"?**

现有设计的问题:用户创建的是"一个评测任务",每次评测是孤立事件,结果散落在各处。用户拿到一堆 task 结果,还是不知道"这颗芯片到底怎么样"。

新设计的核心变化:
- 用户操作的起点是**芯片**,不是"评测任务"
- 评测计划挂在芯片下面,一颗芯片可以有多个评测计划(如初次评测、回归评测)
- 每次评测的结果自动沉淀到**芯片档案**
- 芯片评价报告是最终交付物,汇聚所有评测数据

**Step 2.1 - 选择评测范围**

张工操作:

**为什么用分类树而不是平铺列表?** 因为完整芯片评测可能涉及 200+ 算子,平铺列表会让用户淹没在选项中。分类树让用户可以快速按类选择,也可以展开细选。

**分类体系说明:** 本分类对齐 DeepLink 算子分类方法学(22 大类,400+ 算子),按照 BLAS/Convolution/Norm/Activation/Pooling/Loss/Element-wise/Reduce/Permute/Indexing/Linalg/Interpolate/Distribution/Sort/MISC 等类别组织。这与 DeepLink [AIChipBenchmark](https://github.com/DeepLink-org/AIChipBenchmark) 代码仓库中 `operators/accuracy_test/op_config.py` 定义的 100+ 个算子测例一致,确保 AHVP 产出的评测数据可与 DeepLink 季度评测报告对标。

**Step 2.2 - 配置评测参数**

对已选的评测项,配置详细参数:

**算子功能测试参数(精度验证,对齐 DeepLink 标准):**
- 基准值来源:NVIDIA A100 GPU 上生成的算子输入/输出/梯度数据
- 数据类型:FP32 + FP16(每个算子分别测试两种精度)
- 精度判定标准(对齐 DeepLink AIChipBenchmark):
  - FP32: 绝对误差阈值 `abs_thresh=1e-5`,相对误差阈值 `relative_thresh=1e-4`
  - FP16: 绝对误差阈值 `abs_thresh=1e-3`,相对误差阈值 `relative_thresh=1e-3`
  - 容忍率: 超过阈值的元素比例小于 0.1% (`total_thresh=0.001`)
  - 判定公式: `|output - baseline| <= abs_thresh + relative_thresh * |baseline|`
  - 测试包括前向输出精度 + 反向梯度精度(对 requires_grad=True 的算子)
- 输出: 每个算子的 PASS/FAIL 结果 + CSV 报告
- 参考实现: DeepLink `cuda_op_validate.py` 中的 `result_diff()` 函数

**算子性能测试参数(每个算子可独立配置或批量配置):**
- 输入 Shape:使用预设配置(Small / Medium / Large / Custom)
  - Small: 适合快速验证(如 GEMM 256×256)
  - Medium: 典型场景(如 GEMM 1024×1024,对应 DeepLink gemm_f16.csv/gemm_f32.csv 中的测试参数)
  - Large: 压力测试(如 GEMM 4096×4096)
  - Custom: 自定义 shape 列表(可导入 DeepLink 格式的 CSV 配置)
- 数据类型:FP32 / FP16 / BF16 / INT8(多选)
- Warmup 次数:默认 10
- 测试次数:默认 100
- 超时时间:默认 300 秒/算子
- 性能评分方式:与基准值的时间比值 (score = baseline_time / test_time)
  - score > 1.0: 优于基准芯片
  - score = 1.0: 与基准持平
  - score < 1.0: 低于基准芯片

**模型评测参数:**
- 功能测试(精度): 以 A100 训练结果为基准,对比目标芯片的训练收敛曲线
  - 参考指标: Top-1 Accuracy(分类)、mAP(检测)、mIoU(分割)
  - loss 曲线与基准的 cosine similarity > 0.99 视为通过
- 性能测试(训练): samples/sec、时间到精度 (time-to-accuracy)
- 推理评测参数:
  - 模型规格:Small / Medium / Large
  - Batch Size:1, 4, 8, 16(多选)
  - 推理框架:NumPy / ONNX Runtime / PyTorch(根据环境自动选择)
  - 测试次数:默认 50
  - 大模型推理指标(Phase 2/3): TTFT, TPOT, tokens/sec, 并发处理能力

**为什么要区分 Small/Medium/Large 预设?** DeepLink 评测方案中,算子性能测试使用 CSV 文件定义一组标准测试参数(如 gemm_f16.csv 中定义了多组 m/k/n 参数)。我们将这些参数分级为 Small/Medium/Large 三个预设,快速验证用小参数,完整评测用全量参数。高级用户仍可自定义或导入 DeepLink 格式的 CSV 配置。

### 3.3 数据模型重设计

#### 现有问题诊断

现有数据模型最大的问题是 **`eval_type` 字段混用两个概念**:

```
后端 EvalType 枚举:MODEL / CHIP / FRAMEWORK / OPERATOR
→ 表示的是"评测对象维度"(评的是什么)

前端 EVAL_TYPES:PERFORMANCE / ACCURACY / COMPATIBILITY / STABILITY / GENERAL
→ 表示的是"评测指标类型"(评的是哪个方面)

这两个维度是正交的!
一个算子(OPERATOR)可以同时做性能测试(PERFORMANCE)和精度测试(ACCURACY)。
但现有设计强迫用户只能选一个 eval_type。
```

另一个问题:**评测任务是扁平的**,不属于任何"芯片"或"评测计划"。创建一个任务,跑完就完了,和芯片没有关联。

#### 关键设计决策解释

**1. 为什么 EvaluationTask 用 `test_subject` 替代 `eval_type`?**

原来的 `eval_type` 同时表示"评测对象"(MODEL/CHIP/OPERATOR/FRAMEWORK)和"评测类型"(PERFORMANCE/ACCURACY)两个概念,造成混淆。

新设计:
- `test_subject`:评测对象只有两个值 - `OPERATOR` 或 `MODEL`
  - 为什么不保留 CHIP 和 FRAMEWORK?因为"芯片评测"是由多个算子+模型测试组合而成的(这就是评测计划的作用),不是一个独立的任务类型。"框架评测"同理,它是"在特定框架环境下做算子/模型测试"。
- 评测的"类型维度"(性能/精度/兼容性)不再是枚举字段,而是由评测配置和指标体系来体现

**2. 为什么 Task 必须属于 Plan,Plan 必须属于 Chip?**

这是"以芯片为中心"设计原则的数据层体现。通过强制的层级关系:
- 不会出现"孤儿任务"(不属于任何评测计划的任务)
- 不会出现"孤儿报告"(不关联任何芯片的报告)
- 查询"这颗芯片的所有评测历史"变成简单的外键查询

**3. 为什么 ChipReport 关联的是 Plan 而不是 Task?**

因为芯片评价报告是一次评测计划的综合产物,它聚合了该计划下所有任务的结果。一个报告对应一次完整评测,不是一个单独的任务。

**4. chip_id 冗余存储在 Task 和 Result 中**

虽然可以通过 Task → Plan → Chip 的关系链查到芯片,但评测结果的查询非常频繁(Dashboard、芯片档案都需要),冗余存储 chip_id 可以避免多表 JOIN,提升查询性能。这在数据量增大后尤为重要。

#### 与现有数据模型的差异和迁移思路

| 变化 | 现有 | 新设计 | 迁移方式 |
|------|------|--------|---------|
| **新增 Chip 实体** | 不存在 | 核心实体 | 新增表 `chips` |
| **新增 EvaluationPlan** | 不存在 | 计划层 | 新增表 `evaluation_plans` |
| **EvaluationTask 改造** | 独立存在 | 必须属于 Plan | 现有 task 迁移:创建一个默认 plan + 默认 chip,将现有 task 挂载上去 |
| **eval_type 重设计** | MODEL/CHIP/FRAMEWORK/OPERATOR 枚举 | test_subject: OPERATOR/MODEL | 映射规则:OPERATOR→OPERATOR, MODEL→MODEL, CHIP→忽略, FRAMEWORK→忽略 |
| **TaskTemplate 弱化** | 独立实体 | 变为评测计划创建向导中的"快速配置"选项 | 模板数据迁移为 plan_config 的预设 |
| **Report 改造** | 独立实体,与 task 松散关联 | 改为 ChipReport,强绑定 chip + plan | 现有 report 迁移到 chip_reports 表 |
| **新增 EvaluationResult** | 结果在 task 的 JSONB 中 | 独立实体 | 从现有 task 的结果 JSONB 中拆出 |

#### 报告设计的关键决策

**1. 三层信息架构:总览 → 详情 → 分析**
参考 Azure AI Foundry 的 Index Score → Chart → Detail Table 模式。
- 总览(能力雷达图 + 综合评分)- 10 秒内判断芯片水平
- 详情(算子排行表 + 模型性能表)- 3 分钟了解具体数据
- 分析(瓶颈 TOP3 + 优化建议)- 指导后续行动

**2. 能力摘要用自然语言**
不是只给数字,而是用文字总结优势和短板。非技术人员(如产品经理、采购决策者)也能读懂。

**3. 适用场景推荐**
这是 AHVP 最核心的差异化输出。竞品只给原始数据,AHVP 告诉你"这颗芯片适合什么、不适合什么"。

**4. 瓶颈分析带优化建议**
不只标红"哪个指标差",还给出"为什么差、怎么改"。这对芯片厂商的 Kernel 优化团队极有价值。

**5. MVP 阶段的诚实标注**
报告明确标注"CPU 评测模式",不会误导用户。但核心的评测流程、数据结构、报告框架是完整的,GPU 支持后只需替换评测脚本和增加指标。

### 6.3 评分算法详细设计

#### 维度划分(对齐 DeepLink《AI芯片评测实施方案》五大测试维度)

DeepLink 评测体系将芯片评测分为 5 个测试大项:基本技术规格、软件生态、功能测试(算子功能+模型功能)、性能测试(算子性能+模型性能+通信性能)。AHVP 在报告中映射为以下评分维度:

| 维度 | 英文 Key | 对应 DeepLink 测试项 | 包含的评测项 | MVP | Phase 2/3 |
|------|---------|-----------------|-------------|-----|----------|
| 基本技术规格 | tech_specs | 基本技术规格 (算力/内存/通信/能耗比) | 由芯片注册时填写 | ✅ 手动填写 | 自动检测 |
| 软件生态 | software_eco | 软件生态 (软件栈/开放性) | 框架支持、算子库完整性 | ✅ 手动填写 | 自动检测 |
| 算子功能 | op_accuracy | 功能测试-算子功能 | 100+ 算子 FP32/FP16 精度验证 | ✅ | 扩展算子 |
| 算子性能 | op_speed | 性能测试-算子性能 | GEMM, Conv2d, 长尾算子 | ✅ CPU 模拟 | GPU 实测 |
| 通信性能 | comm_perf | 性能测试-通信性能 | AllReduce 带宽/时延 (OSU/NCCL-test) | ❌ | Phase 2/3 |
| 模型功能 | model_accuracy | 功能测试-模型功能 | 基础模型训练收敛验证 | ✅ MLP | 24个模型 |
| 模型性能 | model_speed | 性能测试-模型性能 | 基础模型训练 samples/sec | ✅ 推理 | 训练性能 |
| 大模型能力 | llm_capability | 大模型评测方案 | LLM 训练/微调/推理 | ❌ | Phase 3 |

MVP 阶段雷达图展示 6 个可用维度(tech_specs, software_eco, op_accuracy, op_speed, model_accuracy, model_speed),通信性能和大模型能力在 Phase 2/3 启用后扩展为 8 维雷达图。

#### 评分公式(参考 DeepLink 基准值对比方式)

**DeepLink 的评分逻辑:** DeepLink 以 NVIDIA A100/H100 的测试数据作为基准值 (baseline),待测芯片的性能与基准值做比值得到得分 (score = baseline_time / test_time)。这种方式简单直观,score > 1.0 表示优于基准。

AHVP 分两个阶段实现:

**MVP 阶段(CPU-only,无基准值):**
```
功能测试评分:
  算子功能得分 = 通过的算子数 / 总测试算子数 × 100
  (每个算子: PASS = FP32通过 AND FP16通过,对齐 DeepLink 精度阈值)

性能测试评分:
  单算子评分 = normalize(1 / latency_mean_ms, 0, max_across_all_operators) × 100
  → 延迟越低,评分越高
  → 归一化到 0-100 范围

维度评分 = weighted_average(该维度下所有评测项评分)

综合评分 = weighted_average(所有维度评分)
→ 默认权重: tech_specs=0.05, software_eco=0.05, op_accuracy=0.25, op_speed=0.25, model_accuracy=0.20, model_speed=0.20
→ 权重可配置(系统设置)
```

**Phase 2/3(GPU,有基准值):**
```
性能评分切换为 DeepLink 基准对比模式:
  算子性能得分 = A100_time / chip_time
  → score > 1.0: 优于 A100
  → score = 1.0: 与 A100 持平
  → score < 1.0: 弱于 A100
  → 转换为百分制: ×100 并 cap 在 0-150 范围

基准值数据来源:
  → DeepLink AIChipBenchmark 提供的 A100 log 日志
  → AHVP 自行在 A100/H100 上跑出的基准值
  → 基准值存储在 baseline_data 表中,按季度更新
```

**为什么选择 1/latency 作为 MVP 基础?** 因为延迟是最通用的性能指标,所有算子都有。吞吐量的单位不统一(ops/s vs QPS),不适合跨算子比较。延迟的倒数本质上就是"单位时间处理能力",和吞吐量正相关。

**为什么 MVP 不做基准值对比?** CPU 模式下没有有意义的基准值。Phase 2/3 GPU 支持后切换为 DeepLink 基准对比模式,以 A100/H100 数据为基准,这也是 DeepLink 季度评测报告的标准做法。

#### 状态判定规则(对齐 DeepLink 精度标准)

```
算子功能测试(精度)状态判定(对齐 DeepLink AIChipBenchmark):
- PASS: FP32 和 FP16 均通过精度验证
  - FP32: |output - baseline| <= 1e-5 + 1e-4 * |baseline|,且失败元素比例 < 0.1%
  - FP16: |output - baseline| <= 1e-3 + 1e-3 * |baseline|,且失败元素比例 < 0.1%
  - 同时验证前向输出精度 + 反向梯度精度 + 参数梯度精度
- WARNING: FP32 通过但 FP16 失败(或反之)
- FAIL: FP32 和 FP16 均失败,或执行报错
- NOT_SUPPORTED: 算子在该芯片上未实现

算子性能测试状态判定:
- PASS: P95/Mean < 1.5x (波动可控) 且 score >= 0.5 (基准的 50% 以上)
- WARNING: P95/Mean >= 1.5x (波动较大) 或 score < 0.5
- FAIL: 执行失败或超时

模型功能测试状态判定:
- PASS: 训练收敛,且与基准 loss 曲线的 cosine_similarity > 0.99
- WARNING: 训练收敛但 cosine_similarity 在 0.95-0.99 之间
- FAIL: 训练不收敛或 cosine_similarity < 0.95

报告瓶颈识别:
- TOP N 瓶颈: 按延迟降序排列,取延迟最高的 3 个算子
- 波动分析: 按 P99/P50 比值降序排列,标出比值 > 2x 的算子
- 精度失败分析: 列出所有功能测试失败的算子,并显示失败的精度级别
  (如"在 relative_thresh=1e-2 级别通过,但在 relative_thresh=1e-4 级别失败")
```

### C.3 DeepLink 精度判定标准详解

DeepLink 算子精度测试基于 `AIChipBenchmark/operators/accuracy_test/` 代码实现:

1. **基准值生成**: 在 NVIDIA A100 上运行 `cuda_ground_truth_gen.py`,生成每个算子的输入、输出、参数和梯度数据(FP32 + FP16)
2. **精度验证**: 在待测芯片上运行 `cuda_op_validate.py`,加载基准输入,对比输出
3. **判定公式**: `|chip_output - baseline_output| <= abs_thresh + relative_thresh * |baseline_output|`
   - 失败元素比例 < 0.1% 则通过
4. **阈值分级**:
   - FP32: abs=1e-5, rel=1e-4 (严格)
   - FP16: abs=1e-3, rel=1e-3 (半精度容忍度更高)
   - 失败时自动尝试宽松阈值 (1e-4/1e-3 → 1e-3/1e-2) 并记录通过级别
5. **测试范围**: 前向计算精度 + 反向梯度精度 + Module 参数梯度精度

### C.4 DeepLink 评测报告格式(数据收集表维度)

DeepLink 每季度评测要求厂商提交《数据收集表》,包含以下维度:

- **预填写表格**: 芯片型号、参测单位、联系人、硬件配置等基本信息
- **基础模型数据收集表**: 每个测试项的结果数据、log 日志相对路径
- **大模型数据收集表**: 训练/微调/推理各项指标
