# 资源管理 PRD v1.0 评审反馈

> **评审人:** 麦克雷（产品经理）  
> **评审日期:** 2026-04-09  
> **评审文档:** docs/product-design/resource-management-prd.md (v1.0, 1296行)  
> **评审结论:** 架构设计优秀，但存在范围过大、优先级不清晰的问题，需要大幅裁剪才能落地

---

## 总体评价

**架构层面做得很好：** 三层资源模型（节点/集群 → 资源池 → 调度）清晰合理，NODE_POOL / K8S_POOL 类型化设计很聪明——一个池只装一种资源，调度不混乱。参考阿里云 ACK / Rancher / EKS 的最佳实践也恰当。数据模型设计完整，API RESTful 规范。集群三种接入方式（kubeconfig / Agent / 云API）覆盖了主流场景。

**主要问题：** 文档把一个 18 周的完整资源管理系统全部塞进了一个 PRD，但对当前项目阶段来说范围过大。我们目前只有 1 个开发机节点在跑评测，大量功能是超前设计。需要明确哪些现在做、哪些先只留设计。

---

## 🔴 必须修改（6 项）

### 1. Phase 1 范围过大，需要进一步聚焦

**问题：** Phase 1 是 "节点管理增强 + 资源池 CRUD"，4 周工期。但当前紧迫的问题是：已有节点管理功能不完善（节点离线了、基础 CRUD 不全）。Phase 1 应该先把已有的做对，再加新功能。

**建议：**
- **Phase 1a（1-2周）：** 修复节点管理基础 — 节点状态同步（心跳→在线/离线）、基础 CRUD 完善、标签管理
- **Phase 1b（2周）：** 资源池 NODE_POOL + 评测任务对接
- 节点审批流程（4.1.1 Agent 自动注册）可以放到 Phase 1b 或更后面——当前还不需要"批量节点部署"
- 节点诊断控制台已经在做了（commit f0821db3），Phase 1 中去掉

### 2. K8s 集群管理（Phase 2）的 MVP 需要大幅削减

**问题：** Phase 2 的 K8s 集群管理覆盖了三种接入方式、命名空间管理、资源配额、证书管理、集群健康检查。6 周工期在只有 1 个 AI agent 开发的情况下非常紧张。

**建议 MVP 削减：**

| 功能 | v1.0 建议 | 理由 |
|------|----------|------|
| 集群注册 - kubeconfig | ✅ 保留 | 最简单最直接 |
| 集群注册 - Agent | ✅ 保留 | 内网场景必需 |
| 集群注册 - 云API | ❌ 推迟到 Phase 4+ | 复杂度高，每个云厂商都不一样，ROI 低 |
| 命名空间 CRUD | ✅ 简化保留 | 只做查看 + 选择，不做在平台创建 namespace |
| 资源配额管理 | ❌ 推迟 | 当前没有多租户，配额管理纯属超前 |
| 证书管理 | ❌ 推迟 | 到期提醒可以，管理不需要 |
| 集群健康检查 | ✅ 保留基础版 | 只做 API Server 可达 + 节点状态 |
| 集群状态 HIBERNATED | ❌ 去掉 | 华为 CCE 特有概念，我们不需要 |

### 3. 调度策略（Phase 4）设计过度

**问题：** 4 种调度策略 + 亲和性/反亲和性 + 优先级抢占 + 自动扩缩容。这是 K8s 调度器级别的复杂度。目前我们的评测任务量很小（一天 10 个左右），这些功能短期完全用不到。

**建议：**
- Phase 1 只做 **least_loaded**（默认最低负载分配），不做其他策略
- 亲和性 / 优先级 / 抢占 整个 Phase 4 推迟到有实际需求时再做
- 文档中保留设计但明确标注 "预留设计，暂不实现"
- 自动扩缩容（HPA）更是遥远的事，建议从路线图中移到 "Future" 部分

### 4. 运维监控（Phase 3）依赖未确定的技术栈

**问题：** 方案假设了 Prometheus + Loki/ES + Redis 等组件，但当前项目只有 PostgreSQL + Spring Boot。引入 Prometheus 和 Loki 意味着运维复杂度大幅增加。

**建议：**
- **Phase 1-2：** 监控数据存 PostgreSQL（跟日志系统一样），够用
- **Phase 3：** 评估是否需要独立监控栈，如果节点 <10 个，PostgreSQL 完全够用
- 不要提前引入重型组件。如果数据量上来了，再迁移到 Prometheus 也来得及
- 告警先做最简单的：定时检查 + 飞书 Webhook 通知。不需要独立的告警规则引擎

### 5. 自愈策略风险高，MVP 应该去掉

**问题：** 4.5.5 设计了 "自动 SSH 重启 Agent" 和 "自动清理 /tmp" 这样的自愈策略。在没有充分测试和灰度的情况下，自动化运维操作风险很高（误判 → 错误操作 → 雪崩）。

**建议：**
- MVP 阶段所有 "自愈" 改为 "告警 + 建议"
- 管理员收到告警后手动处理
- 等系统运行稳定、异常判定准确率足够高后，再逐步开放自动自愈

### 6. 安全设计中 kubeconfig 加密的实现缺细节

**问题：** 9.1 说 "AES-256-GCM 加密" 和 "密钥存于 KMS"。但当前没有 KMS 基础设施，密钥管理方案不完整就上加密，可能导致密钥丢失 → 所有 kubeconfig 无法解密。

**建议：**
- 明确 MVP 阶段的密钥管理方案：环境变量 + 文件（简单可靠）
- 写清楚密钥轮转流程（换密钥时怎么 re-encrypt 已有数据）
- 添加 "忘记密钥" 的恢复方案（答案通常是没有恢复方案 → 需要重新导入 kubeconfig）
- KMS 作为 Phase 3+ 的增强项

## 🟡 建议优化（5 项）

### 7. 节点 "一个节点可加入多个 NODE_POOL" 需要三思

**问题：** 4.3.1 AC-3 说 "一个节点可加入多个 NODE_POOL"。这会导致调度冲突——两个池同时调度任务到同一节点，资源超分。

**建议：**
- MVP 阶段：**一个节点只能属于一个 NODE_POOL**，简单清晰
- 如果确实需要多池复用，必须设计资源预留/锁定机制，防止超分
- 同理，一个集群加入多个 K8S_POOL 问题更小（因为 K8s 本身有资源隔离），但建议也先限制为 1:1

### 8. 前端菜单层级太深

**问题：** 7.1 的导航结构有 3 级（资源管理 → 运维中心 → 告警管理 → 告警规则），加上已有的评测、芯片管理等菜单，整体导航会很臃肿。

**建议：**
- 扁平化：资源管理下只留 "节点管理"、"集群管理"、"资源池"、"运维监控" 四个一级入口
- 告警规则放在运维监控页面的 Tab 里，不单独做菜单项
- 故障记录也整合到运维监控中

### 9. API 设计中节点指标用 SSE 不合适

**问题：** 6.1 中 `GET /api/v1/nodes/{id}/metrics` 标注了 SSE（Server-Sent Events）。对于监控指标来说，前端定时 pull 比 SSE 更合适——因为监控图表是按固定频率刷新的，不需要事件推送。

**建议：**
- 监控指标用 REST + 轮询（30s 间隔，跟日志系统一致）
- WebSocket / SSE 只用在实时日志流上

### 10. 数据模型中 compute_nodes 表改动需要向前兼容

**问题：** 5.1 给 compute_nodes 表加了大量新字段（labels JSONB、hardware_info JSONB、software_info JSONB 等）。当前已有节点数据需要平滑迁移。

**建议：**
- 所有新字段都设为 nullable（NOT NULL 约束不要加）
- 写一个数据迁移脚本，给已有节点填充默认值
- PRD 里加一节 "数据迁移方案"

### 11. pool_resources 表的 CHECK 约束无法直接在 PostgreSQL 中实现

**问题：** 5.1 中 `pool_resources` 表说 "CHECK(pool_type 与 resource_type 匹配)"。但 pool_type 在 resource_pools 表里，不在 pool_resources 表里，PostgreSQL 的 CHECK 约束不能跨表。

**建议：**
- 这个校验在应用层做（Service 层在添加资源时检查 pool_type 与 resource_type 是否匹配）
- 或者在 pool_resources 表冗余一个 pool_type 字段（用触发器或应用层同步）
- PRD 里要写清楚这是应用层校验而非数据库约束

## 📋 优先级重排建议

基于当前项目状态（1 个节点、10 个/天评测任务、1 个 AI agent 开发），建议调整路线图：

| 阶段 | 原 PRD 内容 | 建议调整 | 工期 |
|------|-----------|---------|------|
| **P0** | — | 修复节点基础功能（状态同步、CRUD 完善） | 1 周 |
| **P1** | Phase 1 部分 | 节点标签 + NODE_POOL 资源池 + 评测对接 | 2 周 |
| **P2** | Phase 2 精简 | K8s 集群注册（kubeconfig + Agent 两种）+ K8S_POOL | 3 周 |
| **P3** | Phase 3 精简 | 基础监控面板（PostgreSQL 存储）+ 飞书告警 | 2 周 |
| **Future** | Phase 4 + 剩余 | 高级调度策略、自愈、HPA、云API接入、KMS | 按需 |

总计 P0-P3 约 8 周（对比原路线图 18 周削减了 55%）。

---

## 总结

| 类别 | 数量 | 核心要点 |
|------|------|----------|
| 🔴 必须修改 | 6 | 聚焦 Phase 1、K8s MVP 削减、调度策略推迟、监控不引新组件、去掉自愈、密钥管理 |
| 🟡 建议优化 | 5 | 节点单池、菜单扁平化、SSE→REST、数据迁移、跨表约束 |

**最关键的一句话：** 架构设计是对的，但 18 周的全量路线图需要裁剪到 8 周的 MVP 路线图。先把节点池 + 集群池 + 基础监控跑通，再逐步增强。
