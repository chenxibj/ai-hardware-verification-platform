# 日志系统重构方案评审反馈

> **评审人:** 麦克雷（产品经理）  
> **评审日期:** 2026-04-08  
> **评审文档:** docs/log-system-redesign.md  
> **评审结论:** 整体质量不错，有 4 处必须调整 + 4 处建议优化

---

## 总体评价

方案质量不错。现状分析精准到位（6 个问题全部抓到了），技术选型合理，日志模型设计清晰。前端 LogPanel 的交互设计细节到位（自动滚动+手动暂停、等宽字体、emoji 类型标识）。数据量评估务实，没有过度设计。

主要问题在：Phase 顺序需要调整、几个关键技术细节缺失、全局日志中心只有占位符。

---

## 🔴 必须修改（4 项）

### 1. Phase 1 和 Phase 2 顺序应该互换

**问题：** 现在 Phase 1 是"消除模拟数据 + 扩大显示区域"，但前端删掉模拟数据后，如果 Agent 还是只上报 3 条日志，日志面板就变成了空面板展示 3 行真实数据——体验比假数据更差。

**建议：**
- 先做 Agent 日志上报增强（当前 Phase 2），确保有丰富的真实日志产出
- 再做前端展示改造（当前 Phase 1）
- 或者两个 Phase 并行推进，但部署顺序是 Agent 先上线

### 2. 批量上报 API 缺少幂等设计

**问题：** `/tasks/{taskId}/logs/batch` 如果 Agent 网络抖动重试，同一批日志会写两遍。方案里没有去重机制。

**建议：**
- LogReporter 每批生成一个 `batchId`（UUID）
- 后端按 `(taskId, batchId)` 做唯一约束或幂等检查
- 收到重复 batchId 直接返回 200，不重复写入

### 3. WebSocket 断连重连策略缺失

**问题：** 方案有 WebSocket 连接代码，但没有处理断连场景。生产环境网络抖动、后端重启都会导致 WS 断开，用户会看到日志停止更新但没有任何提示。

**建议：**
- 指数退避重连：1s → 2s → 4s → ... → max 30s
- 重连成功后通过 `sinceSequence` 参数拉取断连期间遗漏的日志
- UI 显示连接状态指示器（绿色=已连接，黄色=重连中，红色=断开）
- 断连超过 30s 自动 fallback 到 HTTP 轮询

### 4. `sequence` 字段设计有缺陷

**问题：** `ALTER TABLE task_logs ADD COLUMN sequence BIGINT DEFAULT 0;` — 所有日志 sequence 默认都是 0，完全失去排序和断点续传的意义。

**建议：**
- 改用 PostgreSQL `GENERATED ALWAYS AS IDENTITY` 或独立 `BIGSERIAL` 确保全局递增
- 这是上次 PRD review 就明确提出的（#5 点：`sequence_no` 生成策略），设计文档需要落实

```sql
ALTER TABLE task_logs ADD COLUMN IF NOT EXISTS sequence BIGINT GENERATED ALWAYS AS IDENTITY;
```

## 🟡 建议优化（4 项）

### 5. LogReporter 的 flush 策略需要补充边界处理

**问题：** 当前策略是"攒够 10 条或每 2 秒"。但评测结束时如果最后一批不够 10 条，且不到刷新时间，这些日志可能丢失。

**建议：**
- 增加 `close()` 方法，评测结束时强制 flush 剩余日志
- 使用 Python `with` 语句或 `atexit.register()` 保证即使异常退出也能 flush

```python
class LogReporter:
    def __enter__(self):
        return self
    
    def __exit__(self, *args):
        self.flush()  # 确保所有日志都上报
```

### 6. 归档方案建议暂不实现

**问题：** 90 天归档到 MinIO + 定时删除——对当前日志量（每天 300 条预估）来说过度了。3 年的数据量也就 30 万条，PostgreSQL 完全无压力。

**建议：**
- Phase 4 的归档改为"预留设计，暂不实现"
- 当数据量超过 100 万条时再启用
- 归档代码作为文档附录保留，不纳入实施计划
- 分区策略同理，当前不需要实现

### 7. 前端日志内存需要设上限

**问题：** `setLogs(prev => [...prev, ...newLogs])` 不限制条数。如果一个 Plan 跑了几千条日志，全部在内存里，低端设备可能卡顿。

**建议：**
- 前端保留最新 2000 条在内存中用于展示
- 更早的日志通过分页 API 懒加载
- 方案里提了 react-window 虚拟滚动，但代码没有实际使用，需要真正集成
- 内存中的日志超过 2000 条后，旧的移出，但提供"加载更早日志"按钮

### 8. Logs 全局日志中心页面缺少实际设计

**问题：** 3.6 节的代码全部是注释占位符，没有实际设计内容。功能列了 6 项（按 Plan 分组、时间范围筛选、日志类型筛选、全文搜索、日志趋势图、无限滚动）但缺少 API 设计、数据结构、页面布局。

**建议：**
- 要么在本文档中补充全局日志中心的具体设计（API + 页面结构 + 交互流程）
- 要么从 Phase 4 拆出来，单独作为 Phase 5，后续再细化

---

## 总结

| 类别 | 数量 | 核心要点 |
|------|------|----------|
| 🔴 必须修改 | 4 | Phase 顺序、幂等设计、WS 重连、sequence 自增 |
| 🟡 建议优化 | 4 | flush 边界、归档暂缓、内存上限、全局日志中心 |

修改 #1（Phase 顺序）和 #4（sequence 自增）影响最大，建议优先处理。其余问题在开发过程中逐步解决即可。
