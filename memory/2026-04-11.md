## Lessons Learned

### 1. 🔴 DaemonSet ≠ Discovery — 两种模式不能混用
**问题**：注册集群时自动部署 DaemonSet，DaemonSet Agent 向平台注册 `k8s-node-01`，同时集群 discovery 注册 `ahvp-k8s/cn-beijing.172.18.188.151`。名字不同 → 去重失败 → 幽灵节点。

**教训**：K8s 节点纳管只能走一种模式。discovery 模式（kubectl get nodes）更可靠，不需要在每个节点上跑 agent。DaemonSet 模式适合需要节点级 agent 执行任务的场景，但当前阶段不需要。

**修复**：`deployCluster()` 跳过 DaemonSet，只走 discovery。

### 2. 🔴 @Transactional + HTTP 调用 = 连接池杀手
**问题**：`recoverTasks()` 带 `@Transactional`，里面调 `dispatchSingleTask()` 做 HTTP POST 到 agent。HTTP 默认无超时 → agent 不可达时连接挂 30s+ → 事务期间 DB 连接不释放 → 10 个连接全堵死 → 整个系统瘫痪（连登录都超时）。

**教训**：
- **永远不要在 @Transactional 方法里做外部 HTTP 调用**
- RestTemplate 必须配超时，不配 = 无限等待
- HikariCP 默认 10 个连接对生产完全不够
- `leak-detection-threshold` 是救命配置，必须开

**修复**：移除外层 @Transactional + HTTP 超时 + 连接池扩大 + 限流

### 3. 🔴 Compose 重建会覆盖容器内修改
**问题**：`docker compose up -d` 重建 backend 时，Dockerfile 的 `COPY kubectl` 复制了源码目录的空文件，覆盖了之前 docker cp 进去的正常 kubectl。

**教训**：
- 源码目录的 binary 文件要用 `.gitignore` 管理，或者在 Dockerfile 里用 `curl` 下载
- `docker cp` 的修改在容器重建时会丢失
- 关键 binary 要有验证步骤（`kubectl version --client`）

### 4. ⚠️ fixedRate 注释和代码不一致
`@Scheduled(fixedRate = 10000) // 每分钟执行一次` — 注释说每分钟，代码是每 10 秒。小问题但说明 code review 不够仔细。

### 5. ⚠️ PENDING 任务堆积是定时炸弹
68 个 PENDING 任务不是突然出现的，是长期积累的。应该有：
- 超时自动取消机制（PENDING 超 24h → CANCELLED）
- 监控告警（PENDING 数 > 阈值 → 告警）
- Dashboard 展示任务状态分布

### 6. ✅ 自动化测试脚本的价值
写了 `k8s-scale-test.sh` 测试脚本，使得多轮测试可重复执行。比手动一步步敲命令效率高得多，也更容易发现 regression。
