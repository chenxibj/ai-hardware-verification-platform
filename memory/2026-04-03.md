### 🔴 关键 Lessons
1. **Sub-agent 改后端代码必须 commit**。未 commit 的代码在 Docker 构建中不可靠（缓存层）
2. **ddl-auto=validate 时，entity 和 DB 必须 100% 匹配**。一个字段名差异就启动失败
3. **Docker 多阶段构建 + COPY src 有缓存陷阱**。文件变了但 Docker 认为没变
4. **docker compose up 会重建所有依赖容器**，导致 docker cp 的修改丢失
5. **永远不要在生产数据库上频繁 ALTER TABLE**。应该先在测试环境验证
6. **修复策略：遇到连环 schema 问题时，先回退到已知可用状态，再渐进式前进**

### 二、Lessons & Learned

**✅ 做得好的：**
- MVP-0 一天内从零到完成，13 个 issue 全关闭，执行力极强
- 白屏事故后立刻建立了完整防御体系（CI/CD + 冒烟测试 + 健康检查），不是修了就完
- 后端 4 小时崩溃修复中保持冷静，最终用"回退到已知可用状态再渐进前进"策略解决
- Sub-agent 使用越来越成熟：直接给代码而非给需求，效率翻倍

**❌ 踩的坑：**
- **白屏 7 小时才发现** — 凌晨部署后没有验证，早上 chenxi 才报告。教训：部署 ≠ 上线。
- **Sub-agent 代码未 commit** — 新实体 15 个 Java 文件只存在工作目录，docker compose --build 丢失。损失 4 小时排查。
- **Entity 和 DB 列名不一致** — sub-agent 建表用一套列名，写 Entity 用另一套，Hibernate validation 直接炸。
- **ddl-auto 切换代价** — validate→update 切换后旧列残留，新旧数据混杂，排查极其痛苦。
- **CI workflow 首次运行覆盖手动修复** — 建了 deploy.yml 后自动触发，反而把刚修好的前端又覆盖了。

**🔧 可以改进的：**
- Sub-agent 后端任务必须强制 commit + 验证启动 + 验证 API 三步检查
- 部署后冒烟验证应该自动化（已通过 #121 解决）
- 凌晨操作后至少做一次浏览器级验证再睡
- 考虑引入 staging 环境，避免直接在"生产"上试错
- 后端 Entity 创建前先 `SELECT column_name FROM information_schema.columns` 确认实际列名
