### 14:15 — /api/eval-logs 500 修复完成
**根因：** Hibernate 6 + PostgreSQL 的 null 参数类型推断 bug。`EvalLogRepository.findFiltered()` 的 JPQL 查询中，当 `search` 为 null 时，Hibernate 6 的 PostgreSQL 方言把参数类型发送为 `bytea` 而非 `text`，导致 PostgreSQL 报错 `function lower(bytea) does not exist`。

**修复：** JPQL → native SQL query，所有 nullable 参数显式 `CAST(:param AS TYPE)`：
- taskId → BIGINT, level → TEXT, search → TEXT, startTime/endTime → TIMESTAMPTZ

**Commit:** `78b6806f`
**验证：** /api/eval-logs → 200 ✅, search filter ✅, stats ✅

### 💡 Lessons Learned
1. **自动化 > 自律 > 承诺（终于有正面案例）** — idle-work-trigger 从设置到首次产出只隔了 1 天。14 天的"明天一定做"不如 1 个 cron job。这是 04-24 终极教训的正面验证。
2. **Hibernate 6 方言升级是隐形地雷** — 从 H5→H6 后 PostgreSQL 方言对 nullable 参数的类型推断行为变了（bytea vs text）。任何框架大版本升级后都应跑全量回归。
3. **NOT NULL 约束应在 schema 设计时就考虑 nullable 场景** — alert_type、chip_id 这类字段，在某些业务路径下确实可以为空（如系统告警无 chip、评测结果暂未绑定芯片），设计时就该 nullable。
