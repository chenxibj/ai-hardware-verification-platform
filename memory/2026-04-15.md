## 遇到的问题

1. **维度 key 双体系（中英文）** — 旧报告用英文 key，新报告用中文。对比页必须双向映射合并。这是个系统性问题，每个接触 dimensionScores 的地方都需要处理。
2. **调度器 dispatch 超量** — 17 个任务一次性标记 RUNNING，Agent 处理不过来。麦克雷加了回退机制（DISPATCHED 超 2 分钟回退 QUEUED），但根本解是分批 dispatch。
3. **Sub-agent 第一次跑 #444 超时** — 后端+前端打包太大，拆成两次后顺利完成。任务粒度控制很重要。

## Lessons Learned

### 维度 key 归一化应在 API 层做
- 前端到处做 key 映射是 bug 温床。正确做法是后端统一输出一套 key（建议全用中文或全用英文+前端统一翻译层），而非让每个前端页面各自映射。
- **行动项：** 提 issue 让后端统一 dimensionScores key 命名。

### 对比功能的正确分层
- 计算逻辑放共享库（comparison.js / ComparisonService），UI 只负责渲染
- 基准切换纯前端重算，不请求后端 — 用户体验秒级响应
- 这个分层设计今天被验证了：5 个 bug 修复都没碰计算逻辑，只改了渲染层

### 一天从 0 到 1 的节奏
- 底层先行（后端 API + 公式库）→ 入口（列表多选）→ 核心页面 → bug 修复
- 严格串行依赖：#444/#445 → #446/#449 → #447/#448 → bug fixes
- 每完成一步就部署+验证，不积压
