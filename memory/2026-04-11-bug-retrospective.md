# Bug 修复质量复盘 — 2026-04-11

## 现象
麦克雷提了 20 个 bug (#361-#380)，4 个 sub-agent 并行修复，全部标记"已修复已验证"。但实际上**多个 bug 并未真正修复**。

---

## 根因分析

### 1. 🔴 验证流程是假的 — "curl 返回 200" ≠ "功能正常"

**这是最核心的问题。**

Sub-agent 的验证标准：
```
curl API → 返回 HTTP 200 → "✅ 已验证"
```

但实际上：
- HTTP 200 只说明没报 500，不代表功能正确
- 没有验证返回的数据是否正确
- 没有验证关联流程是否正常（如：创建任务后能否启动、启动后状态是否正确）
- 没有端到端验证（从前端操作到后端结果）

**本质：我把"编译通过 + API 不报错"当成了"bug 修复"，这是自欺欺人。**

### 2. 🔴 并行 sub-agent 互相踩踏

4 个 agent 同时改同一个代码仓库：
- Agent A 改了 TaskService，build & deploy
- Agent B 也改了 TaskService 的另一处，build & deploy → **覆盖了 A 的改动**
- Agent C `git add -A` 把所有人的改动打包提交 → 未验证合并后是否兼容
- 最终 JAR 里是"最后一个构建的 agent 的版本"

**本质：把代码仓库当成了无锁的共享资源，并发写入必然出问题。**

### 3. 🔴 Sub-agent 不理解业务，只做机械修改

Task prompt 给了"修复 PATCH 返回 500"，sub-agent 就：
1. 加一个 PATCH 端点
2. curl 一下不报错
3. 宣布修复

但它不理解：
- 这个 PATCH 应该支持哪些字段？
- 修改后下游逻辑是否受影响？
- 用户实际操作场景是什么？

**本质：sub-agent 是"代码修改器"不是"问题解决者"。它完成的是 task prompt 的字面要求，不是 bug 的真正修复。**

### 4. 🔴 修 bug 产生新 bug（恶性循环）

典型案例链：
```
#349 发现 127.0.0.1 不可调度 → 加 IP 过滤
#355 加了 172.16/10.x 过滤 → 过滤太狠
#357 172.17.0.1(Docker bridge)被误杀 → 移除过滤
#360 Agent 无限重试 → 加重试限制
```

每次"修复"都只看当前 bug，不考虑全局影响。像打地鼠一样，打一个冒两个。

**本质：缺乏系统性思考，每次修复都是补丁式的，没有从根本上解决问题。**

### 5. 🔴 没有回归测试保护

- 项目有 BDD 测试框架（Playwright），但**修 bug 时从不跑测试**
- 之前 68 个 E2E 测试现在已经废了（compose 重建后环境变了）
- 每次 fix 只验证自己改的那一个接口，不知道有没有搞坏别的

**本质：没有安全网，每次修改都是在走钢丝。**

### 6. ⚠️ Issue 关闭标准太低

当前标准：sub-agent 说修了 + 单接口 curl OK → 关 issue
应有标准：BDD 测试用例通过 + 前端流程走通 + code review → 关 issue

---

## 为什么有这么多 bug？

回看这 20 个 bug 的性质：

| 类型 | 数量 | 例子 |
|------|------|------|
| **端点不存在** | 5 | #362 #368 #372 #378 #379 |
| **字段没传递/没保存** | 4 | #364 #366 #367 #376 |
| **状态机逻辑错误** | 4 | #361 #365 #374 #380 |
| **校验缺失** | 4 | #370 #373 #375 #376 |
| **过度修复导致的新 bug** | 3 | #357 #358 #360 |

其中**端点不存在和字段没传递**占了将近一半 — 这说明**很多功能本来就是半成品**，前端有 UI 但后端没实现或没接好。这不是"bug"，是"功能缺失"被包装成了"已完成"。

**根本原因：之前的冲刺模式（一天 38 个 issue）追求的是速度不是质量。骨架代码关了 issue，但功能链路没跑通。**

---

## 为什么问题反复出现？

1. **同一个问题不同表现**：IP 过滤→连接池→调度失败→任务卡死，是一条因果链，但被拆成 4 个独立 issue 分别"修复"
2. **修复不彻底**：只修了 API 层不修 Service 层，或者只修了后端不修前端
3. **没有从 bug 中提取通用规则**：每次都是 case-by-case 修，没有建立起"芯片名必须唯一""所有创建接口必须校验外键"这样的通用守则

---

## 改进方案

### 短期（立即执行）
1. **修 bug 必须跑完整验证流程**：不是 curl 一下，是走完整的用户操作路径
2. **禁止并行 sub-agent 改同一代码库**：串行执行，或严格按模块拆分不交叉
3. **每个 bug fix 提 PR 而不是直接 push main**：至少有一个 review 环节

### 中期（本周）
4. **恢复 E2E 测试**：修复 Playwright 测试环境，每次部署前跑
5. **建立 API 契约测试**：每个端点有 input/output schema 校验
6. **Sub-agent task prompt 必须包含验证场景**：不是"修复 X"，是"修复 X，并验证 Y 场景能正常工作"

### 长期
7. **Code review 制度化**：菜菜子 review 麦克雷的代码，麦克雷 review 菜菜子的代码
8. **Bug 分类和根因追踪**：每个 bug 追溯到哪个 commit 引入，防止同类问题

## 菜菜子的自我反思

我犯了一个产品经理不该犯的错误：**把关注点放在"issue 数量减少"而不是"功能真正可用"上。**

20 个 issue 全关闭看起来很漂亮，但如果一半没真正修好，那不如不关。这比不修还糟糕 — 因为它给了"一切正常"的假象。

chenxi 说的对，需要停下来想清楚再动手，而不是一味追求速度。

**核心教训：Quality > Speed。做 10 个真正修好的 bug fix，好过 20 个"假装修好"的。**
